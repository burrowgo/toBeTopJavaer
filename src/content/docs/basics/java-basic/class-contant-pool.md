In Java, many people must have heard of the concept of a constant pool. It is also one of the commonly tested topics in interviews. In Java-related interview questions, it is common to assess the interviewee's understanding of the constant pool through questions related to `String`. A few simple `String` interview questions have stumped countless developers. Therefore, the constant pool is a very important concept in the Java system.

Speaking of the constant pool, there are three types of constant pools used in the Java system: the **String Constant Pool**, the **Class Constant Pool**, and the **Runtime Constant Pool**.

This article first introduces what exactly the Class Constant Pool is.

### What is a Class File?

In [Compilation and Decompilation of Java Code](), we introduced the concepts of Java compilation and decompilation. We know that computers only understand 0s and 1s, so the code written by programmers needs to be compiled into a binary format consisting of 0s and 1s before the computer can run it.

As mentioned in [In-depth Analysis of Java Compilation Principles](), to give the Java language good cross-platform capabilities, Java ingeniously provides an intermediate code that can be used on all platforms - ByteCode.

With bytecode, no matter which platform (such as Windows, Linux, etc.), as long as a virtual machine is installed, bytecode can be run directly.

Similarly, having bytecode also decouples the Java Virtual Machine from the Java language. Many people might not understand this. Isn't the Java Virtual Machine just for running the Java language? What does this decoupling refer to?

In fact, the Java Virtual Machine can now support many languages other than Java, such as Groovy, JRuby, Jython, Scala, etc. This support is possible because these languages can also be compiled into bytecode. The virtual machine does not care which language the bytecode was compiled from.

The compiler responsible for compiling bytecode in the Java language is the command `javac`.

> `javac` is the Java language compiler included in the JDK. This tool can compile source files with the `.java` extension into bytecode with the `.class` extension that can run on the Java Virtual Machine.

For example, we have the following simple `HelloWorld.java` code:

```java
public class HelloWorld {
public static void main(String[] args) {
String s = "Hollis";
}
}
```

Generate the class file via the `javac` command:

 javac HelloWorld.java

The generated `HelloWorld.class` file:

![][3]

> How to open a class file in hexadecimal: use `vim test.class`, then in command mode, enter `:%!xxd`.

As you can see, the file above is a Class file, which contains the Java Virtual Machine instruction set, a symbol table, and several other pieces of auxiliary information.

To understand the bytecode above, you need to understand the structure of the Class file. Since this is not the focus of this article, it won't be explained in detail here.

> Readers can see that the first eight letters in the `HelloWorld.class` file are `cafe babe`, which is the magic number of the Class file ([Magic Number in Java]()).

What we need to know is that following the 4-byte magic number in the Class file is the 4-byte version number of the Class file (the 5th and 6th bytes are the minor version number, and the 7th and 8th bytes are the major version number. The version number of the Class file I generated is 52, which corresponds to Java 8. That is, this version of bytecode cannot run on versions below JDK 1.8). Following the version number is the entry to the Class Constant Pool.

### Class Constant Pool

The Class Constant Pool can be understood as a resource warehouse within the Class file. Besides information like class version, fields, methods, and interfaces, the Class file also contains a constant pool (constant pool table), used to store various literals and symbolic references generated by the compiler.

Since the number of constants in different Class files is not fixed, a two-byte constant pool capacity counter is set at the entry of the Class constant pool to record the number of constants in the pool.

![-w697][5]

Of course, there is a simpler way to view the constant pool in a Class file, which is through the `javap` command. For the `HelloWorld.class` above, you can use:

 javap -v HelloWorld.class

The contents of the constant pool are as follows:

![][6]

> As seen in the figure above, there are 16 constants in the constant pool of the decompiled class file. The value of the constant counter in the Class file is `0011`, which is 17 when converted from hexadecimal to decimal.
> 
> The reason is that, unlike Java language habits, the constant pool counter starts from 1 instead of 0. A constant pool count of 17 means there are 16 constants, with index values ranging from 1 to 16.

### What is in the Constant Pool?

After introducing what the Class Constant Pool is and how to view it, let's analyze in depth what is contained within it.

There are two main categories of constants stored in the constant pool: **literals** and **symbolic references**.

### Literals

As mentioned earlier, the Class Constant Pool mainly saves literals and symbolic references. So, what exactly are literals?

> In computer science, a literal is a notation for representing a fixed value in source code. Almost all computer programming languages have literal representations for basic values, such as integers, floating-point numbers, and strings; many also support literal representations for boolean types and character types; some even support literal notation for elements of enumeration types and composite types like arrays, records, and objects.

The above is an explanation of literals in computer science, which isn't very easy to understand. Simply put, a literal refers to a string or numeric value composed of letters, numbers, etc.

Literals can only appear as right-values. A right-value refers to the value on the right side of an equal sign, e.g., in `int a = 123`, `a` is the left-value and `123` is the right-value. In this example, `123` is a literal.

 int a = 123;
 String s = "hollis";

In the code examples above, `123` and `"hollis"` are both literals.

In the `HelloWorld` code at the beginning of this article, `"Hollis"` is a literal.

### Symbolic References

Besides literals, there are also symbolic references in the constant pool. So, what exactly is a symbolic reference?

Symbolic reference is a concept in compilation principles, used in contrast to direct references. It mainly includes three categories of constants:
* Fully qualified names of classes and interfaces
* Field names and descriptors
* Method names and descriptors

This explains why the constant pool mentioned earlier also contains constants such as `com/hollis/HelloWorld`, `main`, and `([Ljava/lang/String;)V`.

### What is the Class Constant Pool Used For?

Having introduced so much about what the Class Constant Pool is, how to view it, and what it stores, a key question remains: what is its actual use?

First, it is clear that the Class Constant Pool is a resource warehouse in the Class file, storing various constants. These constants are defined by developers and need to be used during the program's runtime.

In *Understanding the Java Virtual Machine*, it is stated:

When Java code is compiled with `javac`, there is no "linking" step like in C and C++. Instead, dynamic linking occurs when the virtual machine loads the Class file. That is, the final memory layout information for various methods and fields is not saved in the Class file. Therefore, these symbolic references for fields and methods cannot be used directly by the virtual machine without being converted during runtime to obtain the actual memory entry addresses. When the virtual machine runs, it needs to obtain the corresponding symbolic reference from the constant pool and then resolve and translate it into a specific memory address during class creation or runtime. Content regarding class creation and dynamic linking will be explained in detail during the virtual machine class loading process.

The above passage may seem complex and not easy to understand. In essence, it means: the Class file is a medium for storing constants, and it serves as an intermediate location. When the JVM actually runs, it needs to load the constants from the constant pool into memory.
