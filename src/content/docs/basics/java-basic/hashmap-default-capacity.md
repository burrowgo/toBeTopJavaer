title: "Hashmap Default Capacity"
---

Collections are frequently used in daily Java development, and as a typical K-V data structure, HashMap is certainly familiar to Java developers.

In daily development, we often create a HashMap in the following way:

 Map<String, String> map = new HashMap<String, String>();

However, have you ever wondered what the default capacity of a newly created HashMap is, given that we didn't specify it in the code above? And why?

This article will analyze this question.

### What is Capacity?

In Java, there are two relatively simple data structures for saving data: arrays and linked lists. Arrays are characterized by easy addressing but difficult insertion and deletion; linked lists are characterized by difficult addressing but easy insertion and deletion. HashMap combines arrays and linked lists to leverage the advantages of both, and can be understood as an array of linked lists.

In HashMap, there are two easily confused key fields: `size` and `capacity`. Among them, `capacity` is the capacity of the Map, and `size` is what we call the number of elements in the Map.

A simple analogy makes it easier to understand: HashMap is a "bucket". Capacity is the maximum number of elements the bucket can currently hold, while `size` represents how many elements are already in the bucket.

![-w778][1]

Consider the following code:

 Map<String, String> map = new HashMap<String, String>();
 map.put("hollis", "hollischuang");
 
 Class<?> mapType = map.getClass();
 Method capacity = mapType.getDeclaredMethod("capacity");
 capacity.setAccessible(true);
 System.out.println("capacity: " + capacity.invoke(map));
 
 Field size = mapType.getDeclaredField("size");
 size.setAccessible(true);
 System.out.println("size: " + size.get(map));

Output result:

 capacity: 16, size: 1

In the example above, we defined a new HashMap and put one element into it. Then we printed the `capacity` and `size` via reflection. The capacity is 16, and the number of elements stored is 1.

From the previous example, we found that when we create a HashMap without specifying its capacity, we get a Map with a default capacity of 16. So, where does this capacity come from? And why is it this number?

### Capacity and Hashing

To explain the reason for this default capacity, we first need to know what this capacity is used for.

We know that capacity is the number of "buckets" in a HashMap. When we want to put an element into a HashMap, we need to calculate which bucket it should be placed in through a certain algorithm. This process is called hashing, corresponding to the `hash` method in HashMap.

![-w688][2]

We know that the `hash` method's function is to locate the position of a K-V pair in the linked list array based on its Key. That is, the input of the `hash` method should be an `Object` type Key, and the output should be an `int` type array index. If you were to design this method, how would you do it?

Actually, it's simple: we just need to call the `hashCode()` method of the `Object`, which returns an integer, and then take the modulus of this number with the capacity of the HashMap.

If it were really that simple, setting the capacity of HashMap would be much easier. However, considering efficiency and other issues, the implementation of the `hash` method in HashMap is somewhat complex.

### Implementation of hash

Next, we will introduce the implementation principle of the `hash` method in HashMap. (The following part refers to my article: [The most thorough analysis of hash() in Map on the whole web, no other](). PS: Many articles on the web analyzing the `hash` method of HashMap are "derived" from this article of mine.)

Specifically, it is implemented by two methods: `int hash(Object k)` and `int indexFor(int h, int length)`.

> **hash**: This method mainly converts an `Object` into an integer.
> 
> **indexFor**: This method mainly converts the integer generated by `hash` into an index in the linked list array.

To focus on the main point of this article, let's only look at the `indexFor` method. Let's first look at the implementation details in Java 7 (although there is no such separate method in Java 8, the algorithm for querying the index is the same as in Java 7):

 static int indexFor(int h, int length) {
 return h & (length-1);
 }

The `indexFor` method mainly converts the `hashcode` into an index in the linked list array. The two parameters `h` represents the `hashcode` value of the element, and `length` represents the capacity of the HashMap. So what does `return h & (length-1)` mean?

Actually, it is a modulus operation. The main reason Java uses bitwise operations (`&`) instead of the modulus operator (`%`) is efficiency.

> Bitwise operations (`&`) are much more efficient than the modulus operator (`%`) because bitwise operations directly operate on memory data and do not need to be converted to decimal, thus the processing speed is very fast.

So, why can bitwise operations (`&`) be used to implement the modulus operation (`%`)? The principle is as follows:

 X % 2^n = X & (2^n - 1)

Suppose `n` is 3, then `2^3 = 8`, which is `1000` in binary. `2^3 - 1 = 7`, which is `0111`.

At this time, `X & (2^3 - 1)` is equivalent to taking the last three digits of the binary representation of `X`.

From a binary perspective, `X / 8` is equivalent to `X >> 3`, which is shifting `X` three places to the right. This gives the quotient of `X / 8`, and the part shifted away (the last three digits) is `X % 8`, which is the remainder.

I don't know if you understood the explanation above. If not, it doesn't matter; you just need to remember this trick. Or you can try it with a few examples.

 6 % 8 = 6, 6 & 7 = 6
 
 10 % 8 = 2, 10 & 7 = 2

![][3]

Therefore, `return h & (length-1);` as long as the `length` is guaranteed to be `2^n`, the modulus operation can be implemented.

So, **because bitwise operations directly operate on memory data and do not need to be converted to decimal, bitwise operations are more efficient than modulus operations. Therefore, when HashMap calculates the index to store elements in the array, it uses bitwise operations instead of modulus operations. The prerequisite for making this equivalent replacement is that the capacity of HashMap must be 2^n.**

Then, since it is `2^n`, why must it be 16? Why can't it be 4, 8, or 32?

Regarding the choice of this default capacity, the JDK does not provide an official explanation, and the author hasn't found any valuable information about this on the internet. (If anyone has relevant authoritative information or ideas, feel free to leave a comment.)

According to the author's inference, this should be an **Experience Value**. Since a default `2^n` must be set as the initial value, a trade-off needs to be made between efficiency and memory usage. This value should be neither too small nor too large.

If it's too small, resizing may occur frequently, affecting efficiency. If it's too large, it wastes space and is not cost-effective.

Therefore, 16 was adopted as an experience value.

> In JDK 8, the default capacity is defined as: `static final int DEFAULT_INITIAL_CAPACITY = 1 << 4; // aka 16`. It deliberately writes 16 as `1 << 4` to remind developers that this place must be a power of 2. Interestingly, the **aka 16** in the comment was also added in 1.8.

Next, let's talk about how HashMap ensures that its capacity is always `2^n`. What happens if the user sets it themselves?

Regarding this part, HashMap handles compatibility in two places where its capacity might change: during initialization with a specified capacity and during resizing.

### Initializing with Specified Capacity

When we set the initial capacity through `HashMap(int initialCapacity)`, HashMap does not necessarily directly use the value we pass in. Instead, it calculates a new value to improve hash efficiency. (1->1, 3->4, 7->8, 9->16)

> In JDK 1.7 and JDK 1.8, the timing of HashMap initializing this capacity is different. In JDK 1.8, the capacity is set when the HashMap constructor is called. In JDK 1.7, this operation is delayed until the first `put` operation.

Let's see how the JDK finds the first power of 2 greater than the specified value:

 int n = cap - 1;
 n |= n >>> 1;
 n |= n >>> 2;
 n |= n >>> 4;
 n |= n >>> 8;
 n |= n >>> 16;
 return (n < 0) ? 1: (n >= MAXIMUM_CAPACITY) ? MAXIMUM_CAPACITY: n + 1;

The purpose of the above algorithm is simple: based on the capacity value passed in by the user (`cap` in the code), calculate and return the first power of 2 greater than it.

![][4]

Please pay attention to the changes in the blue font parts in the examples above; you might find some patterns. 5->8, 9->16, 19->32, 37->64 all mainly go through two stages.

> **Step 1**, 5->7
> **Step 2**, 7->8
> 
> **Step 1**, 9->15
> **Step 2**, 15->16
> 
> **Step 1**, 19->31
> **Step 2**, 31->32

Corresponding to the code above, Step 1:

 n |= n >>> 1;
 n |= n >>> 2;
 n |= n >>> 4;
 n |= n >>> 8;
 n |= n >>> 16;

Corresponding to the code above, Step 2:

 return (n < 0) ? 1: (n >= MAXIMUM_CAPACITY) ? MAXIMUM_CAPACITY: n + 1;

Step 2 is relatively simple: it checks the limits and then adds 1 to the value obtained in Step 1.

How to understand Step 1? It actually performs a series of right shifts on a binary number and then performs a bitwise OR with the original value. Its purpose is to set all bits after the first non-zero bit of a binary number to 1.

Take any binary number and apply the formulas above to see the purpose:

 1100 1100 1100 >>> 1 = 0110 0110 0110
 1100 1100 1100 | 0110 0110 0110 = 1110 1110 1110
 1110 1110 1110 >>> 2 = 0011 1011 1011
 1110 1110 1110 | 0011 1011 1011 = 1111 1111 1111
 1111 1111 1111 >>> 4 = 1111 1111 1111
 1111 1111 1111 | 1111 1111 1111 = 1111 1111 1111

Through several unsigned right shifts and bitwise OR operations, we converted `1100 1100 1100` into `1111 1111 1111`. Adding 1 to `1111 1111 1111` gives `1 0000 0000 0000`, which is the first power of 2 greater than `1100 1100 1100`.

Now we have explained the code for Step 1 and Step 2. It can convert a number into the first power of 2 greater than itself.

However, there is a special case where the formulas above don't work: when the number is already a power of 2. If 4 is used in the formulas, you would get 8. But this problem has also been solved. For the specific verification method and JDK's solution, see [The most thorough analysis of hash() in Map on the whole web, no other](), we won't expand on it here.

In short, HashMap uses unsigned right shifts and bitwise OR operations to calculate the first power of 2 greater than the initialization capacity passed in by the user.

### Resizing

In addition to specifying the capacity of HashMap during initialization, its capacity may also change during resizing.

HashMap has a resizing mechanism, which occurs when certain conditions are met. The resizing condition for HashMap is when the number of elements (`size`) in the HashMap exceeds the threshold (`threshold`).

In HashMap, `threshold = loadFactor * capacity`.

`loadFactor` is the load factor, representing how full the HashMap is. The default value is `0.75f`. Setting it to 0.75 has an advantage: 0.75 is exactly 3/4, and `capacity` is a power of 2. Therefore, the product of the two numbers is always an integer.

For a default HashMap, by default, resizing is triggered when its `size` is greater than 12 (16 * 0.75).

The following is a segment from the resizing method (`resize`) in HashMap:

 if ((newCap = oldCap << 1) < MAXIMUM_CAPACITY &&
 oldCap >= DEFAULT_INITIAL_CAPACITY)
 newThr = oldThr << 1; // double threshold
 }

As can be seen from the code above, the size of the table after resizing becomes twice the original. After this step is executed, the resized table will be adjusted. This part is not the focus of this article and is omitted.

Evidently, when the number of elements (`size`) in the HashMap exceeds the threshold (`threshold`), it will automatically resize to twice the original capacity, i.e., from 16 to 32, 64, 128...

Therefore, by ensuring that initial capacities are all powers of 2 and that resizing also doubles the previous capacity, it is guaranteed that the capacity of HashMap is always a power of 2.

### Summary

As a data structure, HashMap needs to perform hash operations during the `put` process to calculate the specific location of the element in the HashMap.

The hash operation process actually involves taking the `hashcode` of the target element's Key and then taking the modulus with the Map's capacity. To improve the efficiency of modulus operations, JDK engineers used bitwise operations instead, which requires the Map's capacity to be a power of 2.

As a default capacity, being too large or too small is inappropriate, so 16 was adopted as a more suitable experience value.

To ensure that the Map's capacity is a power of 2 under any circumstances, HashMap imposes restrictions in two places.

First, if the user specifies an initial capacity, HashMap calculates the first power of 2 greater than that number as the initial capacity.

Additionally, during resizing, the capacity is doubled, i.e., 4 becomes 8, 8 becomes 16.

By analyzing why the default capacity of HashMap is 16, we have delved into the principles behind HashMap. From the code, we can see that JDK engineers have used various bitwise operations to the extreme, trying every possible way to optimize efficiency. It's well worth our learning!
